Parry This!

An intense parry-based combat game with precise mechanics and polished animation.

Game Development Breakdown

Player Controls

Problem

Players need an intuitive control system that allows for complex player expression without overwhelming them, providing "easy to learn, hard to master" mechanics.

Solution

I developed a responsive player control system with several core elements:

  • 8-Directional Movement: Allows players predictable positioning to scan for enemies without being limited in movement options.
  • One-Button Combos: Players can chain up to three attacks, where each subsequent strike has a longer but narrower hitbox, accommodating diverse playstyles.
  • Dash Mechanic: Players disable their hitbox for a quick reposition when parrying isn’t viable, adding a defensive option. Recovery time and a cooldown prevent overuse, encouraging strategic timing.
  • Quality of Life Enhancements: Animation-based input buffers allow smoother combo transitions. Separately, movement adjustments provide players with nuanced control mid-animation, creating a rhythm-based challenge that’s rewarding without sacrificing gameplay flow.

Enemy Types

Problem

I wanted enemies to introduce mechanics organically and encourage aggressive play.

Solution

I designed nine distinct enemy types, each serving a different challenge:

  • Melee and Elite Melee: Introduce the basics of parrying, making melee encounters the foundation for constant movement.
  • Ranged Enemies (Gun and Uzi): Train players in prioritizing targets, requiring precise parries and establishing the concept of “priority targets.”
  • Grenadiers (Normal and Elite): Act as chaos agents, allowing players to deflect grenades when timed precisely, adding layers to positioning and timing challenges.
  • Bazooka Enemies: These final, high-stakes enemies use target-seeking rockets, providing a significant end-game challenge.

Enemy Management Systems

Problem

With increasing enemy types, it was crucial to manage spawns in a way that felt fair, predictable, and challenging without overwhelming players.

Solution

I used an enemy manager to handle spawns, adjusting counts and placements to balance gameplay. Drawing from Game AI Pro, I moved beyond uniform random spawning and implemented Dynamic Weighted Random Ratios, balancing unique runs with fairness.

Priority-Based HFSM Game AI Architecture

Problem

Initial prototypes made enemies predictable and easy to exploit. To manage the varied enemy AI behaviors effectively, I needed a flexible but logical system that supported diverse tactics.

Solution

I chose a custom Priority-Based HFSM architecture, giving the AI flexibility with logical, weighted priorities for actions. This solution included:

  • Architecture: By making HFSM priority-based, it moves closer to industry-standard solutions like Behavior Trees, balancing simplicity with adaptability.
  • Pathfinding: Implemented a vector-based navigation system which weighs vectors based on target or obstacle proximity, allowing agents to pathfind using A* while dynamically avoiding others by adjusting their priority based on proximity to the player character.
  • Variance: Added randomized behavior variance for minor action fluctuations, which prevents repetitive patterns and creates engaging encounters.

Game Feel

Problem

Creating a polished mechanic is built with meaningful sound, visual feedback, and mechanics that engage players, yet without certain tactile hardware limitations, some polish elements weren't achievable.

Solution

Deadlines did not allow for game controller testing, so tactile feedback like rumble wasn't possible. However, I was successful in implementing:

  • VFX: Using SOLID principles, I created management scripts for my 40+ handcrafted VFX, which allowed for simple VFX triggers without changing existing character logic. I expanded with UI indicators for attack time and type, off-screen indicators, and HP icons. This brought a sweet spot between my original idea for diegetic health representation through armor breaks and clear visual communication during chaotic multi-fights.
  • SFX: I recorded and/or mixed various sound effects and composed music loops in FL Studio 12. FMOD is planned for future projects for greater adaptive sound control.
  • Cinematic Effects: Effects like time dilation and controlled screen shakes intrinsically reward the player and reinforce impactful moments.

Performance

Problem

As development progressed, the game faced significant performance issues, particularly with graphical rendering and memory usage.

Solution

  • Visual Optimization: I optimized materials, for example - bullets move so fast that an unlit shader looked identical to a lit one on them, but saved performance time. In order to eliminate the complexity of an opaque and transparent render queue, I introduced custom dithering shaders to introduce door-to-door transparency.
  • Memory Management: I reduced garbage collection time from 5ms to 0.02ms, replacing GetComponent calls with serialized fields or singletons to minimize CPU overhead.
  • Asynchronous Scene Loading: My loading screen is a reusable scene for any scene-to-scene transition, making scene transitions seamless and optimized.

Visual Identity

Problem

To create a cohesive art style, I avoided asset packs or generative AI, designing 3D and 2D assets from scratch. Only a few floor and wall textures were sampled from AmbientCG. While a good start, I needed something more distinct than just a low-poly 3D game.

Solution

  • Custom Visual Assets: I aimed for a stylized, handcrafted aesthetic that reflects the game’s tone and themes. I ensured the UI is always animated and includes smooth transitions for a polished finish.
  • UI Optimization: Since I wanted to stylize every component, I used a UI Camera instead of the default Overlay, allowing for post-processing effects and ensuring visual cohesion across all game elements.
